# Reusable MVP Template (Any Platform)

Use this template to quickly create a focused extension concept from marketplace feedback.

---

## 1) Opportunity snapshot

- Platform: [Chrome Web Store / Edge Add-ons / Firefox Add-ons / Product Hunt / AppSumo / etc.]
- Category: [category]
- Target user: [persona]
- Core job-to-be-done: [single sentence]
- Trigger moment: [when user feels pain]
- Current workaround: [how users cope now]

---

## 2) Evidence pack

### Quantitative
- Avg rating for top competitor set: [value]
- Total ratings signal: [value]
- Total review sample analyzed: [value]
- Complaint cluster frequency: [top 3 percentages]

### Qualitative
- Representative complaint #1: "[...]"
- Representative complaint #2: "[...]"
- Representative complaint #3: "[...]"

---

## 3) Problem statement

Users who [persona/context] struggle to [job], because existing tools [specific failure].
This causes [time loss / frustration / risk / missed outcomes].

---

## 4) MVP scope

### Core promise
[one sentence: what your extension does better than alternatives]

### Must-have features (max 3)
1. [feature]
2. [feature]
3. [feature]

### Not in MVP (explicit)
- [feature deferred]
- [feature deferred]

### Trust baseline
- Permissions shown clearly in product copy
- Local-only storage or explicit data policy disclosure
- One-click disable/reset path

---

## 5) UX flow

1. Install extension
2. First-use onboarding (<= 30 seconds)
3. First successful core action
4. Repeat action prompt
5. Feedback collection trigger

---

## 6) Metrics plan

### Activation
- `% users who complete first core action within 1 day`

### Retention
- `D1 retention`
- `D7 retention`

### Value
- `Core action count per active user`

### Sentiment
- `In-app thumbs up/down`
- `Review sentiment shift after releases`

---

## 7) Event schema (minimum)

- `install`
- `onboarding_completed`
- `core_action_started`
- `core_action_completed`
- `error_shown`
- `feedback_submitted`

---

## 8) Build plan (2-week default)

### Week 1
- Day 1–2: wireframes + architecture
- Day 3–4: core feature implementation
- Day 5: QA + bug pass

### Week 2
- Day 1: instrumentation + onboarding polish
- Day 2: closed beta release
- Day 3: collect feedback
- Day 4: fix top blockers
- Day 5: public launch

---

## 9) Release checklist

- [ ] Clear value proposition in listing
- [ ] Permission rationale visible
- [ ] Privacy page linked
- [ ] Basic onboarding complete
- [ ] Error states covered
- [ ] Metrics/events verified

---

## 10) Post-launch loop

- Pull new reviews weekly
- Re-score opportunities monthly
- Keep a living backlog tied to validated complaints
